CONTEXT

什么是 Skill，为什么要用 Skill

03 / 13
简单类比
Skill 就像给 AI 配了一套“岗位说明书 + SOP + 工具箱 + 质检清单”
提示词告诉 AI 这次要做什么；Skill 告诉 AI 这类事以后都应该怎么稳定做好。
01
岗位说明书
说明这个 Skill 负责哪类任务，什么时候该触发，什么时候不该用。
对应name、description、适用 / 不适用范围。
02
SOP 流程
把专家经验拆成清晰步骤，让 AI 不用临场猜下一步怎么做。
对应工作流、输入输出、执行顺序。
03
工具箱
把模板、规范、示例和脚本沉淀下来，下次遇到同类任务可以直接复用。
对应references、assets、scripts。
04
质检清单
明确什么算完成、怎么检查通过，让交付从“感觉可以”变成“确实达标”。
对应验收标准、检查命令、失败回写。

SCENARIO

什么事情可以变成一个 Skill？

04 / 13
01
可重复
不是只做一次的临时需求，而是未来还会遇到同类问题，可以沉淀成稳定做法。
02
稳定流程
已经有相对固定的工作流、步骤顺序和输入输出，AI 可以按流程执行，而不是每次重新猜。
03
有特定知识
需要参考规范、模板、素材或者脚本，才能稳定做出符合场景要求的结果。
适合沉淀的例子
旅行计划
每次出行都要整理目的地、预算、交通、行程和注意事项。
大促资源位
制作不同尺寸的大促资源位，统一活动主题、版式规则和输出规格。
批量图标生成
批量生成规格、风格一致的图标，保证尺寸、线宽、圆角和命名统一。
消息通知
反复写群通知、活动提醒、会议提醒，需要讲清时间、地点、动作和截止时间。

STANDARD

优秀 Skill 的五大核心判断

05 / 13
01
触发清楚
看什么name 和 description 能让 AI 判断什么时候该用、什么时候不用。避免只写能力名，缺少适用场景。
02
流程可执行
看什么有明确步骤、输入、输出、命令或检查点。避免只讲原则，不告诉 AI 下一步怎么做。
03
能复用
看什么模板、规范、示例和脚本能被下次任务继续使用。避免把一次性经验写成不可维护的长文。
04
渐进式披露
看什么主流程完整，补充资料按任务需要再打开。避免一开始读取所有 references，干扰判断重点。
05
有验收标准
看什么明确什么算完成，以及如何检查通过。避免只要求“高质量”，但没有可验证证据。

STRUCTURE

一个 Skill 的推荐结构

06 / 13
核心文件
SKILL.md 只放入口、流程和验收
它负责让 AI 判断“什么时候用”、知道“怎么做”、明确“什么算完成”。
my-skill/ ├─ SKILL.md ├─ references/ ├─ assets/ └─ scripts/
SKILL.md（必有）
Skill 的入口说明：触发场景、工作流程、输出格式和验收标准。
适合放：name、description、核心步骤、禁区和检查要求。
references（可有）
给 AI 读的说明书：规范、口径、案例、检查清单。
适合放：版式库、业务规则、评分口径、示例说明。
assets（可有）
给 AI 使用的素材或底稿：模板、图片、Logo、字体、样例文件。
适合放：PPT 模板、Word 模板、品牌素材、可复用图片。
scripts（可有）
给 AI 调用的自动化动作：校验、转换、批处理、尺寸检查。
适合放：validate、render、export、clean 等重复命令。

PROGRESSIVE DISCLOSURE

渐进式披露：按需加载资源

08 / 13
按需加载资源
先读主流程
SKILL.md 先说明任务流程和判断规则，让 AI 知道当前任务应该怎么走。
再按场景选资料
只在需要某个 Step 或场景时，打开对应 reference 文件。
降低干扰，提升稳定性
避免资料一次性塞满上下文，让 AI 聚焦当前问题和必要规则。

CHECKLIST

生成前必须问清的四类输入

09 / 13
输入维度
需要问清什么
影响什么
用户目标
用户最终要交付什么，听众是谁，时间和质量要求是什么。
决定 Skill 的输出口径与文字密度。
任务边界
哪些场景适用，哪些场景应该拒绝、澄清或切换方法。
决定触发条件与禁区说明。
可复用资产
是否已有模板、脚本、参考案例、设计规范、历史样例。
决定 assets、references 和 scripts 的组织方式。
验收标准
如何判断完成：命令通过、截图合格、格式正确、内容无占位。
决定最终检查清单和自动化验证。

USE

使用 Skill 后，要复盘它是否真的变稳

11 / 13
阶段
检查问题
优化方向
创建前
这个任务是否反复出现？是否有稳定流程？是否需要模板、知识或脚本？
优先选择高频、边界清楚、有样例的场景。
编写时
name 是否短、description 是否写清触发、SKILL.md 是否只保留核心流程？
详细规则拆到 references，模板素材拆到 assets，重复动作拆到 scripts。
使用时
它是自然触发、明确点名，还是与其他 Skill 组合完成复杂任务？
观察触发是否准确，必要时补充关键词和不适用场景。
复盘时
输出是否比不用 Skill 更稳定？哪一步让 AI 误解？哪些资料应该补充？
把失败案例写回流程、禁区、样例或校验清单。

PRACTICE

课堂小作业：8 张 Skill 任务卡

12 / 13
CARD 01
需求澄清官
requirement-clarifier
任务把模糊需求拆成追问问题。
输出目标 / 关键问题 / 风险点。
测试输入我们想做一个会员成长体系。
通关不少于 6 个问题，只提问不直接给方案。
CARD 02
空状态文案师
empty-state-copy
任务生成产品空状态文案。
输出标题 / 说明 / 主按钮 / 次按钮。
测试输入搜索商品没有结果。
通关标题不超 12 字，文案自然，引导下一步。
CARD 03
周报润色器
weekly-report-polisher
任务把流水账整理成结果导向周报。
输出成果 / 进行中 / 风险阻塞 / 下周计划。
测试输入跟首页改版、开会、看竞品，埋点未定。
通关不编数据，能识别埋点未定风险。
CARD 04
用户反馈翻译器
feedback-translator
任务把用户吐槽翻译成产品问题。
输出原意 / 可能问题 / 影响范围 / 建议验证。
测试输入这个页面太烦了，找订单要点半天。
通关区分事实和推测，验证方法可操作。
CARD 05
竞品拆解助手
competitor-breakdown
任务根据竞品页面描述拆解可借鉴点。
输出结构 / 亮点 / 可借鉴 / 不建议照搬。
测试输入会员页含会员卡、积分任务、权益和进度条。
通关借鉴点有适用条件，照搬风险不少于 2 条。
CARD 06
多角色翻译机
role-translator
任务把同一需求翻译给不同角色。
输出给产品 / 设计 / 研发 / 测试。
测试输入订单列表里可以快速申请售后。
通关角色关注点不同，且不改变原需求。
CARD 07
复盘提纲生成器
retro-outline
任务把项目描述整理成复盘提纲。
输出目标 / 结果 / 做得好的 / 问题 / 下次改进。
测试输入活动效果一般，社群通知，页面赶，未测试。
通关不美化失败，没有数据就写缺少数据。
CARD 08
提示词修理工
prompt-fixer
任务把模糊提示词改成可执行提示词。
输出原问题 / 改写后提示词 / 改写理由。
测试输入帮我写个活动方案。
通关包含角色、任务、约束、输出格式。
